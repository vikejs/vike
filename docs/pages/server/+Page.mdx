The `+server.ts` (or `+server.js`) file defines your server entry point and is the recommended way to integrate Vike with your server for universal deployment.

It should export a default object with a `fetch` property, making your server compatible with various platforms.

Note that this file is optional — if you don't declare one, Vike internally provides a default server entry point automatically.


## Integrating Vike with your server

Vike provides a middleware that can be used with any server framework. It handles the rendering of your pages.

## Settings

```js
// +server.ts

export default {
  // ...

  prod: {
    // Server port, defaults to 3000. It's ignored in Cloudflare and Vercel Edge (there
    // isn't any server in serverless deployment).
    port: 3000,

    hostname: 'localhost',

    // Called once the server is accepting connections
    onReady(server) {
      // `server` is an instance of srvx Server. See https://srvx.h3.dev/guide/server

      // This hook will not be called on Cloudflare and Vercel Edge (there
      // isn't any server in serverless deployment)
      console.log('Server is ready.')
    },

    // Called once the server is created
    onCreate(server) {
      // `server` is an instance of srvx Server. See https://srvx.h3.dev/guide/server

      // This hook will not be called on Cloudflare and Vercel Edge (there
      // isn't any server in serverless deployment)
    },

    // HTTPS Options (Node.js/Bun/Deno only)
    tls: {
      // For example SSL/TLS key and certificate for HTTPS
      key: fs.readFileSync('/etc/letsencrypt/live/example.com/privkey.pem'),
      cert: fs.readFileSync('/etc/letsencrypt/live/example.com/fullchain.pem'),
      // optional
      passphrase: 'passphrase'
    },

    // Will serve static files automatically (default)
    static: true,
    // Can be disabled with
    static: false,
    // Or serve from a custom folder
    static: './public'
  }
}
```

<blockquote>
  The above settings only apply to production. For development, use [Vite's server settings](https://vite.dev/config/server-options) instead.

  🚧 Server settings are separate on purpose — values usually differ between dev and prod. If you have a use case where settings share the same value in dev and prod, then [reach out](https://github.com/vikejs/vike/discussions) and we'll create server settings that apply to both dev an prod.
</blockquote>

## Standalone

Using [standaloner](https://github.com/nitedani/standaloner/?tab=readme-ov-file#with-vite), the build output directory ([`dist/`](https://vite.dev/config/build-options.html#build-outdir)) contains everything needed for deployment. This means that, in production, only the `dist/` directory is required (you can remove `node_modules/` and skip `$ npm install`).

> Skipping `node_modules/` massively reduces disk usage in production, which can be important, for example to [reduce the size of Docker images](https://github.com/photon-js/photon/issues/42#issuecomment-3549657194).

```ts
// vite.config.ts

import { defineConfig } from 'vite'
import standaloner from 'standaloner/vite'

export default defineConfig({
  plugins: [standaloner()]
});
```

## HMR

If you change a server file, the server code is automatically updated: the next HTTP response will be generated by the latest code. No full server reload is required.

If HMR isn't what you want (for example if you modify the database connection) you can manually trigger a full server reload by pressing `r + enter`.


## Compression

You can compress all server responses using the universal [compress](https://universal-middleware.dev/middlewares/compress) middleware:

```shell
npm i @universal-middleware/compress
```

```ts
// +server.ts

import compress from '@universal-middleware/compress'
import { Hono } from 'hono'
import vike from '@vikejs/hono'

const app = new Hono()
// Add middleware here
vike(app, [compress()]) // [!code ++]
```
