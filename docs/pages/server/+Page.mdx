The `+server.js` file defines the server entry (Hono, Express.js, ...). For example with Hono:

```ts
// +server.ts

import type { Server } from 'vike/types'
import { Hono } from 'hono'
import vike from '@vikejs/hono'

const app = new Hono()
vike(app)

export default {
  fetch: app.fetch
} satisfies Server
```

It's optional — if you don't define `+server.js`, Vike uses an built-in fallback server.

> Vike uses [Universal Deploy](https://github.com/universal-deploy/universal-deploy) for deploying `+server.js` to any deployment provider.

> Defining a `+server.js` file is the recommended way to integrate Vike with a server, but you can also use <Link href="/renderPage" /> instead.

## Settings

```js
// +server.ts

export default {
  // Production settings
  prod: {
    // Server port, defaults to 3000. It's ignored in Cloudflare and Vercel Edge (there
    // isn't any server in serverless deployments).
    port: 3000,

    hostname: 'localhost',

    // Called once the server is accepting connections.
    // - The `server` object comes from srvx, see https://srvx.h3.dev/guide/server
    // - This hook isn't called on Cloudflare and Vercel Edge (there isn't
    //   any server in serverless deployments).
    onReady(server) {
      console.log('Server is ready.')
    },

    // Called once the server is created
    // - The `server` object comes from srvx, see https://srvx.h3.dev/guide/server
    // - This hook isn't called on Cloudflare and Vercel Edge (there isn't
    //   any server in serverless deployments).
    onCreate(server) {
      console.log('Server created.')
    },

    // HTTPS Options (Node.js/Bun/Deno)
    tls: {
      // For example SSL/TLS key and certificate for HTTPS
      key: fs.readFileSync('/etc/letsencrypt/live/example.com/privkey.pem'),
      cert: fs.readFileSync('/etc/letsencrypt/live/example.com/fullchain.pem'),
      // Optional
      passphrase: 'passphrase'
    },

    // Serve static files automatically
    static: true, // Default value
    // Can be disabled:
    static: false,
    // Or serve from a custom folder:
    static: './public'
  }
}
```

<blockquote>
  The above settings only apply to production. For development, use [Vite's server settings](https://vite.dev/config/server-options).

  🚧 Server settings are separate because values usually differ between dev and prod. If you have a use case where settings share the same value in dev and prod, then [reach out](https://github.com/vikejs/vike/discussions) and we'll create server settings that apply to both dev an prod.
</blockquote>

## Standalone

Using [standaloner](https://github.com/nitedani/standaloner/?tab=readme-ov-file#with-vite), the build output directory ([`dist/`](https://vite.dev/config/build-options.html#build-outdir)) contains everything needed for deployment. This means that, in production, only the `dist/` directory is required (you can remove `node_modules/` and skip `$ npm install`).

> Skipping `node_modules/` massively reduces disk usage in production, which can be important, for example to [reduce the size of Docker images](https://github.com/photon-js/photon/issues/42#issuecomment-3549657194).

```ts
// vite.config.ts

import { defineConfig } from 'vite'
import standaloner from 'standaloner/vite'

export default defineConfig({
  plugins: [standaloner()]
});
```

## HMR

If you change a server file, the server code is automatically updated: the next HTTP response will be generated by the latest code. No full server reload is required.

If HMR isn't what you want (for example if you modify the database connection) you can manually trigger a full server reload by pressing `r + enter`.


## Compression

You can compress all server responses using the universal [compress](https://universal-middleware.dev/middlewares/compress) middleware:

```shell
npm i @universal-middleware/compress
```

```ts
// +server.ts

import compress from '@universal-middleware/compress'
import { Hono } from 'hono'
import vike from '@vikejs/hono'

const app = new Hono()
// Add middleware here
vike(app, [compress()]) // [!code ++]
```
